# 胶水编程 (Glue Coding)

**来源**: https://github.com/tukuaiai/vibe-coding-cn/blob/develop/assets/documents/principles/fundamentals/胶水编程.md

**学习时间**: 2026-03-15

---

## 核心思想

> **能抄不写，能连不造，能复用不原创。**

胶水编程不是技术，是一场革命——从「生成」到「连接」的根本性转变。

### 三大痛点的解法

| 传统 Vibe Coding 痛点 | 胶水编程解法 |
|---------------------|------------|
| 🎭 **AI 幻觉** - 生成不存在的 API、错误逻辑 | ✅ **零幻觉** - 只使用已验证的成熟代码 |
| 🧩 **复杂性爆炸** - 项目越大越失控 | ✅ **复杂性归零** - 每个模块都是久经考验的轮子 |
| 🎓 **门槛过高** - 需要深厚编程功底 | ✅ **门槛消失** - 只需描述"连接方式" |

### 范式转移

```
传统编程：    人写代码
Vibe Coding：  AI 写代码，人审代码
胶水编程：    AI 连接代码，人审连接
```

---

## 核心理念

### 什么不该做

- ❌ 不再让 AI 从零生成代码（幻觉的根源）
- ❌ 不再重复造轮子（复杂性的根源）
- ❌ 不再要求你理解每一行代码（门槛的根源）

### 什么该做

- ✅ 只复用成熟的、经过生产验证的开源项目
- ✅ AI 的唯一职责：理解你的意图，将模块连接起来
- ✅ 你的唯一职责：描述清楚"输入是什么，输出要什么"

## 架构模型

```
┌─────────────────────────────────────────────────────────┐
│ 你的业务需求                                            │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│ AI 胶水层 (Glue Layer)                                  │
│                                                         │
│ "我理解你要做什么，让我把这些积木连起来"                 │
│                                                         │
└─────────────────────────────────────────────────────────┘
                          │
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼
  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
  │ 成熟模块 A   │   │ 成熟模块 B   │   │ 成熟模块 C   │
  │ (10 万+ ⭐)   │   │ (生产验证)   │   │ (官方 SDK)   │
  └─────────────┘   └─────────────┘   └─────────────┘
```

### 三要素

| 要素 | 定义 | 示例 |
|------|------|------|
| **实体 (Entity)** | 成熟的开源项目、官方 SDK、久经考验的库 | `python-telegram-bot`, `pandas` |
| **连接 (Link)** | AI 生成的胶水代码，负责数据流转和接口适配 | 50 行数据转换代码 |
| **功能 (Function)** | 你描述的业务目标 | "Telegram 消息经 GPT 处理后存 PostgreSQL" |

## 五步工作流

```
1. 明确目标
   └─→ "我要实现 XXX 功能"

2. 寻找轮子
   └─→ "有没有成熟的库/项目已经做过类似的事？"
   └─→ 让 AI 帮你搜索、评估、推荐

3. 理解接口
   └─→ 把官方文档喂给 AI
   └─→ AI 总结：输入是什么，输出是什么

4. 描述连接
   └─→ "A 的输出要变成 B 的输入"
   └─→ AI 生成胶水代码

5. 验证运行
   └─→ 跑通 → 完成
   └─→ 报错 → 把错误扔给 AI，继续粘
```

## 实战案例

**需求**: 实时获取 Polymarket 数据，分析后推送到 Telegram

### 传统做法
- 从零写爬虫、写分析逻辑、写 Bot
- **3000 行代码，2 周时间**

### 胶水做法
```
轮子 1: polymarket-py (官方 SDK)
轮子 2: pandas (数据分析)
轮子 3: python-telegram-bot (消息推送)

胶水代码：50 行
开发时间：2 小时
```

## 项目组织建议

```
your-project/
├── /services        # 业务逻辑层
├── /libs            # 封装层
├── /third_party     # 第三方开源仓库（不可修改）
└── /glue            # 胶水代码（你唯一需要写的）
```

**关键原则**: 开源仓库作为第三方依赖，**绝对不可修改**。

## 胶水代码的职责

你写的代码只承担：

- **组合** - 把多个模块组装在一起
- **调用** - 按顺序触发各模块
- **封装** - 统一接口风格
- **适配** - 数据格式转换

---

## 对比：传统开发 vs 胶水编程

| 维度 | 传统开发 | 胶水编程 |
|------|---------|---------|
| 功能实现方式 | 自己写 | 复用开源 |
| 工作量 | 大 | 小得多 |
| 成功率 | 不确定 | 高 |
| 速度 | 慢 | 极快 |
| 错误率 | 容易踩坑 | 使用成熟方案 |
| 重点 | "造轮子" | "组合轮子" |

## 与递归自优化系统的关联

### 生成器空间的映射

在递归自优化框架中：

| 组件 | 胶水编程对应 |
|------|------------|
| 生成器 G | AI 胶水层（负责连接模块） |
| 产出 P | 胶水代码 + 模块配置 |
| 优化器 O | 运行验证 + 错误反馈 |
| 元生成器 M | 更新 AI 的"连接策略" |

### 不动点的意义

当 AI 学会"如何正确连接某类模块"后：
- 类似需求的胶水代码质量趋于稳定
- 不再犯同样的接口适配错误
- 达到该领域的**生成能力不动点**

## 我的理解

### 本质

胶水编程是**工程智慧的最高体现**——用最少的原创代码，撬动最大的生产力。

> "The best code is no code at all. The second best is glue code."

### 为什么 AI 时代尤其适合

1. **AI 擅长读文档** - 快速理解多个模块的输入输出
2. **AI 擅长转换** - 数据格式适配是 AI 的强项
3. **AI 不擅长发明** - 从零生成代码容易幻觉

胶水编程让 AI 做擅长的事，避开不擅长的事。

### 与 OpenClaw 的关联

咱们现在的多 agent 系统，本质上也是胶水编程：
- **Daisy** = 数据采集模块的胶水
- **Luna** = 微博监控模块的胶水
- **Alfred** = 协调各 agent 的超级胶水

但问题是：这些"胶水"还不够模块化，没有复用成熟组件。
