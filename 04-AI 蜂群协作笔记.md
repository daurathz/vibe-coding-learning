## 核心思想

> **AI 不再是孤立的，而是可以互相感知、通讯、控制的集群。**

### 范式转移

```
传统模式：人 ←→ AI₁, 人 ←→ AI₂, 人 ←→ AI₃ (人是瓶颈)

蜂群模式：人 → AI₁ ←→ AI₂ ←→ AI₃ (AI 自主协作)
```

### 三大能力

| 能力 | 实现方式 | 效果 |
|------|---------|------|
| **感知** | `capture-pane` | 读取任意终端内容 |
| **控制** | `send-keys` | 向任意终端发送按键 |
| **协调** | 共享状态文件 | 任务同步与分工 |

## 技术架构

### tmux 架构

```
┌─────────────────────────────────────────────┐
│                 tmux server                  │
├─────────────────────────────────────────────┤
│  Session 0                                   │
│  ├── Window 0:1 [AI-1] ◄──┐                 │
│  ├── Window 0:2 [AI-2] ◄──┼── 互相可见/控制 │
│  ├── Window 0:3 [AI-3] ◄──┤                 │
│  └── Window 0:4 [AI-4] ◄──┘                 │
└─────────────────────────────────────────────┘
```

### 数据流

```
┌─────────┐  capture-pane   ┌─────────┐
│  AI-1   │ ◄───────────────│  AI-4   │
│ (执行)  │                 │ (监控)  │
└─────────┘  send-keys      └─────────┘
     ▲      ───────────────►     │
     │                           │
     └───────── 控制流 ──────────┘
```

## 命令参考

### 信息获取

```bash
# 列出所有会话
tmux list-sessions

# 列出所有窗口
tmux list-windows -a

# 列出所有窗格
tmux list-panes -a
```

### 内容读取

```bash
# 读取指定窗口内容（最近 N 行）
tmux capture-pane -t <session>:<window> -p -S -<N>

# 示例：读取会话 0 窗口 1 最近 100 行
tmux capture-pane -t 0:1 -p -S -100

# 批量读取所有窗口
for w in $(tmux list-windows -a -F '#{session_name}:#{window_index}'); do
  echo "=== $w ==="
  tmux capture-pane -t "$w" -p -S -30
done
```

### 发送控制

```bash
# 发送文本 + 回车
tmux send-keys -t 0:1 "ls -la" Enter

# 发送确认
tmux send-keys -t 0:1 "y" Enter

# 发送特殊按键
tmux send-keys -t 0:1 C-c        # Ctrl+C
tmux send-keys -t 0:1 C-d        # Ctrl+D
tmux send-keys -t 0:1 Up         # 上箭头
```

### 窗口管理

```bash
# 创建新窗口
tmux new-window -n "ai-worker"

# 创建并执行命令
tmux new-window -n "ai-1" "kiro-cli chat"

# 关闭窗口
tmux kill-window -t 0:1
```

## 协作协议

### 状态文件结构

```bash
/tmp/ai_swarm/
├── status.log      # 全局状态日志
├── tasks.json      # 任务队列
├── locks/          # 任务锁
│   ├── task_001.lock
│   └── task_002.lock
└── results/        # 结果存储
    ├── ai_1.json
    └── ai_2.json
```

### 状态日志格式

```bash
[HH:MM:SS] [窗口 ID] [状态] 描述

# 示例
[08:15:30] [0:1] [START] 开始处理 data-service 代码审计
[08:16:45] [0:1] [DONE] 完成代码审计，发现 5 个问题
[08:16:50] [0:2] [WAIT] 等待 0:1 审计结果
[08:17:00] [0:2] [START] 开始修复问题
```

### 协作规则

| 规则 | 描述 | 实现 |
|------|------|------|
| **先查后做** | 开始前扫描其他终端 | `capture-pane` 全扫 |
| **避免冲突** | 相同任务只做一次 | 检查 locks 目录 |
| **主动救援** | 发现卡住主动帮助 | 检测 `[y/n]` 等待 |
| **状态广播** | 完成后通知其他 AI | 写入 status.log |

### 冲突处理

```bash
# 获取锁
if [ ! -f /tmp/ai_swarm/locks/file_x.lock ]; then
  echo "$TMUX_PANE" > /tmp/ai_swarm/locks/file_x.lock
  # 执行任务
  rm /tmp/ai_swarm/locks/file_x.lock
fi
```

## 架构模式

### 对等模式 (P2P)

```
┌─────┐     ┌─────┐
│ AI₁ │◄───►│ AI₂ │
└──┬──┘     └──┬──┘
   │           │
   ▼           ▼
┌─────┐     ┌─────┐
│ AI₃ │◄───►│ AI₄ │
└─────┘     └─────┘
```

**特点**: 所有 AI 平等，互相监控  
**适用**: 简单任务，无明确依赖

### 主从模式 (Master-Worker)

```
        ┌──────────┐
        │ AI-Master│
        │  (指挥官) │
        └────┬─────┘
             │ 分发/监控
    ┌────────┼────────┐
    ▼        ▼        ▼
┌──────┐ ┌──────┐ ┌──────┐
│Worker│ │Worker│ │Worker│
│ AI-1 │ │ AI-2 │ │ AI-3 │
└──────┘ └──────┘ └──────┘
```

**特点**: 一个指挥，多个执行  
**适用**: 复杂项目，需要统一协调

### 流水线模式 (Pipeline)

```
┌─────┐    ┌─────┐    ┌─────┐    ┌─────┐
│ AI₁ │───►│ AI₂ │───►│ AI₃ │───►│ AI₄ │
│分析 │    │设计 │    │实现 │    │测试 │
└─────┘    └─────┘    └─────┘    └─────┘
```

**特点**: 任务串行流转  
**适用**: 有明确阶段的工作流

### 混合模式

```
           ┌──────────┐
           │ AI-Master│
           └────┬─────┘
                │
    ┌───────────┼───────────┐
    ▼           ▼           ▼
┌──────┐   ┌──────┐    ┌──────┐
│分析组 │   │开发组 │    │测试组 │
├──────┤   ├──────┤    ├──────┤
│AI-1  │   │AI-3  │    │AI-5  │
│AI-2  │   │AI-4  │    │AI-6  │
└──────┘   └──────┘    └──────┘
```

**特点**: 分组协作 + 统一调度  
**适用**: 大型项目，多团队并行

## 实战案例

### 案例 1: 多服务并行开发

**场景**: 同时开发 data-service、trading-service、telegram-service

**窗口分配**:
```
0:1 - AI-Master (指挥官)
0:2 - AI-Data (data-service)
0:3 - AI-Trading (trading-service)
0:4 - AI-Telegram (telegram-service)
```

**指挥官提示词核心**:
```
每 2 分钟执行一次扫描：
for w in 2 3 4; do
  echo "=== 窗口 0:$w ===" 
  tmux capture-pane -t "0:$w" -p -S -20
done

发现问题时：
- 卡住等待 → send-keys 确认
- 报错 → 分析并给出建议
- 完成 → 记录并分配下一任务
```

### 案例 2: 代码审计 + 自动修复

**流程**:
```
AI-1 (审计):
1. 扫描代码，输出问题列表
2. 每发现一个问题，写入 /tmp/ai_swarm/issues.log

AI-2 (修复):
1. 监控 issues.log
2. 读取新问题
3. 自动修复
4. 标记完成
```

### 案例 3: 7x24 值守

```bash
# 每个 AI 的监控逻辑
while true; do
  for w in $(tmux list-windows -a -F '#{window_index}'); do
    output=$(tmux capture-pane -t "0:$w" -p -S -5)
    
    # 检测卡住
    if echo "$output" | grep -q "\[y/n\]"; then
      tmux send-keys -t "0:$w" "y" Enter
      echo "已帮助窗口 $w 确认"
    fi
    
    # 检测错误
    if echo "$output" | grep -qi "error\|failed"; then
      echo "窗口 $w 出现错误，需要关注"
    fi
  done
  sleep 30
done
```

```markdown
## 🎖️ AI 集群指挥官协议

- `Waiting` `Blocked` → 任务阻塞
```

## 最佳实践

### 初始化流程

```bash
# 1. 创建共享目录
mkdir -p /tmp/ai_swarm/{locks,results}
touch /tmp/ai_swarm/status.log

# 2. 开启 tmux 会话
tmux new-session -d -s ai

# 3. 创建多个窗口
tmux new-window -t ai -n "master"
tmux new-window -t ai -n "worker-1"
tmux new-window -t ai -n "worker-2"

# 4. 在每个窗口启动 AI
tmux send-keys -t ai:master "kiro-cli chat" Enter
tmux send-keys -t ai:worker-1 "kiro-cli chat" Enter

# 5. 发送蜂群提示词
```

### 命名规范

```bash
# 会话命名
ai          # AI 工作会话
dev         # 开发会话
monitor     # 监控会话

# 窗口命名
master      # 指挥官
worker-N    # 工作节点
data        # data-service 专用
trading     # trading-service 专用
```

### 安全建议

1. **不要自动确认危险操作**：rm -rf、DROP TABLE 等
2. **设置操作白名单**：只允许特定命令
3. **保留操作日志**：记录所有 send-keys 操作
4. **定期人工检查**：不要完全无人值守

## 风险与限制

### 已知风险

| 风险 | 描述 | 缓解措施 |
|------|------|---------|
| 误操作 | AI 发送错误命令 | 设置命令白名单 |
| 死循环 | AI 互相触发 | 添加冷却时间 |
| 资源竞争 | 同时修改同一文件 | 使用锁机制 |
| 信息泄露 | 敏感信息被读取 | 隔离敏感会话 |

### 技术限制

- tmux 必须在同一服务器
- 无法跨机器协作（需要 SSH）
- 终端输出有长度限制
- 无法读取密码输入（隐藏字符）

## 我的理解

### 核心突破

蜂群协作的本质是**打破 AI 的孤立状态**。

传统多 agent 系统：
- 每个 agent 是黑盒
- 只能通过中心调度器通信
- 无法感知彼此的状态

蜂群模式：
- 所有 agent 共享一个"视觉层"（tmux 窗口）
- 可以直接读取其他 agent 的输出
- 可以直接干预其他 agent 的执行

### 为什么是 tmux

1. **标准化接口** - `capture-pane` / `send-keys` 是通用协议
2. **实时性** - 终端输出即时可见
3. **持久化** - 会话可以后台运行
4. **轻量级** - 无需额外基础设施

### 与胶水编程的关联

蜂群协作是胶水编程在**agent 层面**的体现：
- 每个 agent 是一个成熟模块
- Master agent 是胶水层
- 协作协议是连接方式
